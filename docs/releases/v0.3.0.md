# rat v0.3.0 release notes

Release scope: Go CLI and embedded runtimes since `v0.2.2`.

This is a reliability and hardening release. It keeps the everyday `rat py`, `rat run`, `rat look`, and shared-kernel workflows the same, while tightening unsafe edges around secrets, runtime names, daemon health, generic runtimes, and concurrent starts.

## Highlights

- Secrets from saved runtimes no longer travel through daemon process arguments.
- Runtime names are now validated and generated project names are slugged into path-safe identifiers.
- `rat <lang> 2` starts only the requested second instance instead of also starting the base instance.
- Daemon health checks now parse and validate real MCP responses instead of accepting any HTTP response.
- Generic JSON runtimes now delegate `ctl(status)` to the child runtime, so `rat status -v` can show runtime-provided versions and health.
- State-file access now uses a cross-process lock to reduce concurrent-start races.

## Added

### `rat status --json`

`rat status` can now emit machine-readable JSON:

```sh
rat status --json
```

This includes the same enriched status fields used by the human display, including runtime state, PID, port, memory, idle time, clients, last caller, and runtime version when available.

### Structured MCP `run` results

The MCP `run` tool now returns structured content alongside the legacy text output. Clients can read output, error, execution count, duration, variable count, and status separately instead of parsing the trailing `✓ 10ms` / `✗ 10ms` hint from text.

The text fallback is preserved for existing clients.

### Full Python inspection support over MCP

The MCP `look` tool now accepts `full: true` for untruncated Python inspection output where supported. Python docstring output is also less aggressively truncated by default, with `RAT_DOCSTRING_MAX_CHARS` available to cap it.

### State file locking

Rat now serializes state-file operations with an inter-process lock file next to `state.yaml`. This makes concurrent CLI invocations safer, especially around port allocation and daemon startup.

## Changed

### Safer saved env/options handling

Saved runtime env vars and structured options are no longer passed to daemon-managed `rat serve` processes as `--env KEY=VALUE` or `--opt KEY=VALUE` arguments.

Instead:

- saved env vars are passed through the child environment;
- saved structured options are passed through an internal bootstrap env var and then removed before runtime subprocess startup;
- manual/explicit `rat serve --env` and `rat serve --opt` behavior remains available for app-builder and debugging workflows.

### Redacted secret display

User-facing output now redacts secret-bearing values:

- `rat add --env TOKEN=...` prints `env:TOKEN=<redacted>`;
- env checks in generic installers are redacted;
- sensitive-looking option keys such as `token`, `api_key`, `password`, `secret`, and `auth` are redacted in display output.

Ordinary non-secret options such as model names remain visible.

### Runtime name validation

Runtime names are now path-safe identifiers. They must start with a letter or number and may contain only:

```txt
letters, numbers, ., _, -, @
```

Examples now rejected:

```txt
../evil
py/evil
py\evil
my runtime
.hidden
-runtime
_runtime
```

Generated project runtime names are slugged, so a project directory like `my project!` becomes a runtime such as:

```txt
py@my-project
```

### More honest `rat reset` help text

`rat reset` no longer promises that the runtime process is preserved. The command clears the runtime namespace, but many runtimes restart their interpreter/session under the hood to guarantee a clean state.

Use `rat restart <runtime>` when you specifically want an explicit daemon restart.

### Generic JSON runtime status delegation

For generic JSON runtimes, `ctl(status)` now sends a `status` request to the runtime subprocess when the runtime is idle. This lets runtimes report their own state and runtime version.

Built-in R, Slack, Jupyter, and test runtimes now support this path.

Custom generic JSON runtimes should implement:

```json
{"op":"status"}
```

and return one of:

```json
{"text":"idle\nruntime_version: MyRuntime 1.2.3"}
```

or:

```json
{"state":"idle"}
```

## Fixed

### `rat <lang> 2` no longer starts the base runtime first

Previously, `rat py 2` could start `py@project` before starting `py@project.2`. It now resolves the requested instance first and starts only that instance.

### Invalid `rat look` flag combinations no longer auto-start kernels

`rat look <runtime> --at x --code y` now fails validation before resolving or starting a runtime.

### `rat install py` rechecks dependencies after creating `.venv`

When `rat install py` creates a new `.venv`, it now recomputes IPython/Jedi availability against the venv interpreter before deciding whether to install dependencies. This fixes cases where system Python had IPython/Jedi but the newly-created venv did not.

### Input cancellation no longer leaves Python blocked forever

If a client cancels or fails an MCP elicitation request for `input()`/stdin, rat now sends `ctl(cancel)` to the runtime. This avoids leaving the shared kernel stuck waiting for input.

### Stronger daemon health checks

Daemon startup/reuse now validates MCP responses rather than accepting any HTTP response from `/mcp`. Health checks now reject:

- non-2xx HTTP responses;
- JSON-RPC error envelopes;
- MCP tool errors;
- empty status results;
- status text beginning with `error`.

This makes stale or unhealthy daemons less likely to be reused silently.

### Safer daemon port allocation

Daemon startup now allocates a port and records the started process while holding the state-file lock. This reduces races where two simultaneous starts could choose the same port.

## Compatibility notes

### Runtime names

Existing runtime names that contain spaces, path separators, or begin with `.`, `-`, or `_` are no longer valid. Re-add those runtimes with a path-safe name:

```sh
rat remove '<old-name>' --yes
rat add py-new-name /path/to/project --lang py
```

Generated language/project names are slugged automatically.

### Custom generic JSON runtimes

Custom JSON runtimes should add support for `op: "status"`. Without it, `rat status -v` and daemon health checks may report the runtime as unhealthy or unable to provide status.

### Secrets

Saved env vars are still persisted in rat's private state file, but they are no longer echoed to terminal output or passed through daemon argv. No migration is required.

## Release checklist

1. Confirm tests pass:

   ```sh
   go test ./...
   ```

2. Confirm the release tag. This note assumes `v0.3.0` because runtime-name validation is a compatibility change from `v0.2.2`.

3. Tag and push:

   ```sh
   git tag v0.3.0
   git push origin v0.3.0
   ```

4. The existing GitHub Actions + GoReleaser workflow builds the release from tags matching `v*`.
